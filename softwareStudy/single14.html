알고리즘의 복잡도

 알고리즘의 소요 시간을 정확히 평가할 수는 없으므로, 자료의 수 n이 증가할 때 시간이 증가하는 대략적인 패턴 시간 복잡도라는 이름으로 나타내게 된다. 이를 Big-O 표기법으로 주로 나타낸다. 예를 들어 입력 자료의 크기 n에 대하여 $O(n)$의 시간복잡도를 가진 알고리즘은 대략 크기 n에 비례하는 수의 연산을 수행한다고 보면 된다. 알고리즘의 종류에 따라 시간 복잡도의 평가기준도 다양하다. 일반적으로는 $O(n)$의 시간 복잡도를 가지면 좋은 알고리즘으로 취급하며, $log n$의 지수승이 붙는 정도로 막으면$\mathcal{O}(n \log n)$ 등 매우 바람직한 결과이다. $\mathcal{O}(n^3)$ 정도만 돼도 큰 자료수에선 급격히 느려진다.검색 알고리즘의 경우 $\mathcal{O}(1)$이나 $\mathcal{O}(\log n)$의 시간 복잡도를 가지는 알고리즘을 선호하고, 정렬 알고리즘의 경우 $\mathcal{O}(n \log n)$의 시간 복잡도를 가지는 알고리즘을 선호한다. 여기에 각각 해당되는 자료구조/알고리즘에는 해시 테이블과 퀵 정렬 등이 있다.

 공간 복잡도 역시 앞서 언급한 시간 복잡도처럼 Big-O 표기법을 주로 사용한다. 현실에서는 시간 복잡도보다 중요도는 떨어지는데, 시간이 적으면서 메모리까지 지수적으로 증가하는 경우는 없기 때문이고, 다항 시간 내에서 나타나는 메모리 문제들은 보통 알고리즘 자체보다는 알고리즘의 구현에서 발생하는 문제이기 때문이다. 다만 동적 계획법의 경우에는 메모리가 상당히 많이 필요하기 때문에 (즉, 공간복잡도가 크기 때문에) 입력 값의 범위가 넓으면 사용하지 못하는 경우가 많다. 또한 임베디드, 펌웨어 등 하드웨어 환경이 극도로 한정되어 있을 경우 공간 복잡도도 상당히 중요하게 보게된다. 이론적으로는 n에 대한 다항식만큼의 공간으로도 해결이 안 되는 정말 어려운 문제들을 생각하기도 한다.

Big-O 표기법

알고리즘의 효율성을 표기해주는 표기법이다. 알고리즘의 효율성은 데이터 개수(n)가 주어졌을 때 덧셈, 뺄셈, 곱셈 같은 기본 연산의 횟수를 의미한다. 보통 알고리즘의 시간 복잡도와 공간 복잡도를 나타내는데 주로 사용 된다. (각각 시간 효율성,공간(메모리) 효율성을 의미한다.) 시간과 공간 복잡도를 나타내는 방법으로는 점근 표기법이라고 해서 빅오(Big-O), 빅오메가(big-Ω),빅세타(big-Θ) 표기법이 있다.

 "시간과 공간 복잡도는 모두 Big-O 표기법을 사용한다.위의 정리된 내용을 바탕으로 시간 복잡도와 공간 복잡도를 계산해보자."

 시간 복잡도를 계산할 때, 입력의 크기가 정해졌을 때, 알고리즘이 사용되는 연산의 횟수로 표현한다. 실제 시간이 아닌, 연산의 횟수로 나타내는 이유는 컴퓨터마다 기본 연산의 실행 시간이 다르기 때문이다.
 예를 들어, 슈퍼 컴퓨터는 두 비트 덧셈, 곱셈, 비교, 교환 등의 기본 연산에 10의 -11승초(10 pico sec.) 소요된다. 개인용 컴퓨터는 기본 연산에 10의 -8승초(10 nano sec.)로 소요된다. (슈퍼 컴퓨터의 1000배)
입력 크기가 n이라고 할 때 다음과 같이 표기한다.
•  $T(n) - n^2 + 2n + 1 = O(n^2)$ : 최고차항만 나타낸다.
•  $T(n) - 2n - O(n)$ : 최고차항의 계수는 제외한다.
다음 알고리즘을 가지고 시간 복잡도를 구해보자. 알고리즘에 단계별 연산 횟수는 주석과 같고 총 연산 횟수는 4n+5이다. 그러므로 이 알고리즘의 시간 복잡도는 다음과 같다.
• $ T(n) = 4n + 5 = O(n)$

 공간 복잡도도 대략적인 계산이 필요하다. 기존 알고리즘 문제는 예전에 공간 복잡도도 고려되어야할 때 만들어진 경우가 많다. 그래서 기존 알고리즘 문제에 시간 복잡도뿐만 아니라, 공간 복잡도 제약 사항이 있는 경우가 있다. 또한, 기존 알고리즘 문제에 영향을 받아서, 면접시에도 공간 복잡도를 묻는 경우도 있다. 공간 복잡도 계산은 알고리즘에서 실제 사용되는 저장 공간을 계산하면 된다.
 예를 들어, n! 팩토리얼을 구하면 n의 값에 상관없이 변수 n, 변수 fac, 변수 index만 필요하다. 최종적으로 공간 복잡도는 $O(1)$이 나오게 된다.
 다음 알고리즘을 보고 공간 복잡도를 계산해보자. 알고리즘은 4개의 변수를 사용하고 있다.
•  int arr[n] : 4*n byte (입력공간)
•  int n : 4 byte (입력공간)
•  int x : 4 byte (보조공간)
•  int i : 4 byte (보조공간)
 결국, 총 4n + 12 에 메모리를 요구한다. 메모리가 입력 값에 따라 선형적으로 증가하기 때문에 공간 복잡도는 $O(n)$이 된다.

알고리즘의 복잡도 표현

• 풀기 쉬움(tractable)
  - 최악의 경우 다항식 시간 복잡도를 갖는 알고리즘으로 풀 수 있는 문제이다.
  - 적당히 큰 입력의 문제에 대해 상대적으로 짧은 시간에 해를 구할 수 있다.
  - 하지만 차수가 매우 크다면, 매우 오랜 시간이 걸린다.
• 풀기 어려움(intractable)
  - 최악의 경우 다항식 시간 복잡도를 갖는 알고리즘으로 풀 수 없는 문제이다.
  - 입력의 크기가 작은 경우에 조차 최악의 경우 문제를 푸는 데 엄청나게 많은 시간이 걸린다.
  - 실제 응용에서는 정답을 찾는 대신, 근사해를 찾는다.
  - 근사해를 찾는 빠른 알고리즘이 존재하며, 정답과 큰 차이가 없다는 것이 보장되는 경우도 있다.
• 풀 수 없음(unsolvable)
  - 해결하기 위한 알고리즘이 존재하지 않다는 것이 증명된 문제이다.
  - 대표적인 예로는 Alan Turing의 정지 문제가 있다.

P vs NP

• NP(nondeterministic ploynomial time) 클래스
  - 최악의 경우 다항식 시간에 푸는 알고리즘이 존재하지 않는 것으로 생각된다. 즉, 풀기 어렵다.
  - 하지만 일단 답이 주어지면 정답인지를 다항식 시간 안에 점검할 수 있는 문제들의 집합
  - 풀기 쉬운 문제들은 P 클래스에 속한다.
• NP 완전 문제(NP-complete problems)
  - 여기에 속한 문제 중 하나라도 최악의 경우 다항식 시간에 풀린다면 NP에 속한 모든 문제가 다항식 시간 알고리즘에 풀린다.
• P vs NP 문제(P versus NP problem)
  - 다항식 시간에 해를 점검하는 것이 가능한 문제의 클래스인 NP가 풀기 쉬운 문제의 클래스인 P와 같은 지 묻는 것이다.
  - 가장 유명한 미확인 문제 중 하나이며 7개의 Millennium Prize Problem 중 하나이다.
